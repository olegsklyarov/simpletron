# Simpletron

Учебный программный симулятор гипотетического компьютера **Simpletron** и его машинного языка **Simpletron Machine Language (SML)**. Задача описана в книге Deitel & Deitel «C++ How to Program» ([страница издания на deitel.com](https://deitel.com/c-plus-plus-how-to-program-10-e/)) в виде «Building Your Own Computer».

Simpletron — это машина с **аккумулятором** и **памятью на 100 слов**. Каждое слово — целое число в диапазоне `-9999..+9999`. Программа на SML — это последовательность таких слов, где каждое 4-значное положительное слово трактуется как **инструкция**: первые две цифры — код операции, последние две — операнд (адрес ячейки памяти).

## Архитектура симулятора

Состояние машины описывается структурой `simpletron_t` ([`simpletron.c`](./simpletron.c)):

```c
typedef struct simpletron {
    int memory[100];           // 100 слов памяти, каждое в диапазоне -9999..+9999
    int accumulator;           // регистр-аккумулятор для арифметики
    int instructionCounter;    // счётчик инструкций (адрес следующей команды)
    int instructionRegister;   // копия текущей выполняемой инструкции
} simpletron_t;
```

Загрузка программы:

- Симулятор читает слова из файла, переданного в `argv[1]`, по одному в строке.
- Слово `-99999` — **маркер конца программы**, дальше файл не читается.
- Допустимо не более **100 слов** до маркера (по числу ячеек памяти).
- При синтаксической ошибке во входе, переполнении или превышении лимита симулятор печатает диагностику в `stderr`, делает дамп состояния и завершается с ненулевым кодом.

Выполнение:

- `instructionCounter` стартует с `0`, после каждой команды увеличивается на 1.
- `instructionRegister = memory[instructionCounter]`, `opcode = IR / 100`, `operand = IR % 100`.
- Команды ветвления модифицируют `instructionCounter` напрямую.
- Аккумулятор после `ADD` / `SUBTRACT` / `MULTIPLY` нормализуется в кольцо `-9999..+9999` (поведение «overflow wraparound»).

### Набор инструкций (SML)

| Код | Мнемоника        | Описание                                                            |
|-----|------------------|---------------------------------------------------------------------|
| 10  | `READ`           | Считать слово из stdin в `memory[operand]`.                         |
| 11  | `WRITE`          | Вывести `memory[operand]` в stdout.                                 |
| 20  | `LOAD`           | `accumulator <- memory[operand]`.                                   |
| 21  | `STORE`          | `memory[operand] <- accumulator`.                                   |
| 30  | `ADD`            | `accumulator += memory[operand]`, результат сворачивается в `-9999..+9999`. |
| 31  | `SUBTRACT`       | `accumulator -= memory[operand]`, с тем же кольцом.                 |
| 32  | `DIVIDE`         | `accumulator /= memory[operand]` (целочисленно, деление на ноль — ошибка). |
| 33  | `MULTIPLY`       | `accumulator *= memory[operand]`, с тем же кольцом.                 |
| 40  | `BRANCH`         | Безусловный переход: `instructionCounter <- operand`.               |
| 41  | `BRANCH_NEG`     | Переход, если `accumulator < 0`.                                    |
| 42  | `BRANCH_ZERO`    | Переход, если `accumulator == 0`.                                   |
| 43  | `HALT`           | Завершить выполнение, вернуть 0.                                    |

Ошибки времени выполнения (попытка деления на ноль, неизвестный опкод, ошибка чтения слова со stdin, выход `instructionCounter` за пределы памяти) приводят к ненулевому коду возврата и дампу состояния в `stderr`.

## Сборка и запуск

Требуется компилятор C11 (`cc`) и Python 3.

```bash
make           # собрать ./simpletron
make test      # собрать и прогнать тесты
make -B        # пересборка
```

Запуск симулятора напрямую:

```bash
./simpletron путь/к/программе.txt
```

- Программа читается из файла, имя которого передано первым (и единственным) аргументом.
- Ввод для инструкций `READ` подаётся в `stdin`.
- Вывод инструкций `WRITE` идёт в `stdout`, диагностика и дамп — в `stderr`.
- Коды возврата: `0` — успешный `HALT`; `1` — ошибка (загрузки, ввода, выполнения).

Пример: программа складывает два числа из stdin и выводит сумму.

```bash
cat > add.sml <<'EOF'
+1007
+1008
+2007
+3008
+2109
+1109
+4300
-99999
EOF

printf '2\n3\n' | ./simpletron add.sml
# 5
```

## Тестовый фреймворк

Тесты — это файлы `*.stest` под каталогом `tests/` (любая вложенность). Каждый файл **полностью описывает один кейс**: SML-программа, ввод и ожидаемый результат. Идея формата — как у [phpt](https://php.github.io/php-src/miscellaneous/writing-tests.html): секции с построчным разбором. Раннер — [`run_tests.py`](./run_tests.py), запускается через `make test`.

Секция начинается строкой `--имя--` (имя из строчных латинских букв и цифр, дефисы допускаются). Тело секции — всё до следующей строки `--...--` или до конца файла. Файл в UTF-8; ожидаемый stdout/stderr сравниваются с выводом процесса **побайтово**.

Для каждого `.stest` раннер:

1. Записывает тело секции `--program--` во **временный файл** в системном каталоге временных файлов, запускает `./simpletron <этот_файл>`, затем **удаляет** временный файл (в том числе при ошибке или таймауте).
2. Подаёт тело секции `--stdin--` в stdin процесса.
3. Если есть секция `--expect-timeout--`, ожидается, что процесс **не** завершится за 5 секунд (остальные ожидания не проверяются).
4. Иначе сравнивает stdout с телом `--expect-stdout--`, stderr с телом `--expect-stderr--` (если секции нет — ожидается пустой stderr), код возврата с первой строкой `--expect-exit--` (если секции нет — ожидается `0`).

Жёсткий лимит — **5 секунд** на запуск. Если процесс не завершился и таймаута не ждали — кейс провален.

### Секции

| Секция               | Обязательность | Смысл |
|----------------------|----------------|--------|
| `--program--`        | да             | Текст SML-программы (как в файле для `simpletron`), обычно завершается строкой `-99999`. |
| `--stdin--`          | да             | Ввод для программы (может быть пустым). |
| `--expect-stdout--`  | да, кроме кейса с таймаутом | Ожидаемый stdout. |
| `--expect-stderr--`  | нет            | Ожидаемый stderr; по умолчанию пусто. |
| `--expect-exit--`    | нет            | Ожидаемый код возврата (число, обычно одна строка); по умолчанию `0`. |
| `--expect-timeout--` | нет            | Если секция есть, ожидается отсутствие завершения за 5 с; тело не используется. |

### Артефакты раннера

| Файл           | Назначение |
|----------------|------------|
| `NN.run`       | Фактический stdout последнего запуска `NN.stest` (рядом с файлом теста). |
| `NN.run.err`   | Фактический stderr последнего запуска. |

Файлы `*.run` и `*.run.err` в `.gitignore`.

### Создание нового теста

1. Создайте каталог, например `tests/my-feature/`.
2. Добавьте `tests/my-feature/01.stest` с секциями `--program--`, `--stdin--`, `--expect-stdout--` и при необходимости `--expect-stderr--`, `--expect-exit--`.
3. Для бесконечного цикла укажите секцию `--expect-timeout--` вместо проверки вывода.
4. Прогоните `make test`. Подробный вывод провалов: `python3 run_tests.py -v`.

### Запуск раннера вручную

```bash
python3 run_tests.py             # обычный прогон
python3 run_tests.py -v          # подробный вывод по проваленным тестам
python3 run_tests.py --tests-dir tests --simpletron ./simpletron
```

При успешном прогоне всех тестов раннер завершается с кодом `0`, иначе — `1`.

## Структура репозитория

```
.
├── simpletron.c         # симулятор
├── Makefile             # сборка (-Wall -Wextra -Wpedantic -std=c11 -O2)

├── tests/               # тестовые наборы
│   ├── add/             # сложение двух чисел
│   ├── subtract/        # вычитание

│   ├── max/             # максимум из двух (BRANCH_NEG)
│   ├── overflow-wrap/   # переполнение аккумулятора кольцом
│   ├── infinite-loop/   # бесконечный цикл (expect_timeout)
│   └── negative-too-many-words/  # >100 слов в программе
└── .github/workflows/ci.yml      # build + test в CI
```
